# 1단계: 아이디어 검증 가이드

> **목표:** "이 문제가 실제로 존재하는가?"를 확인한다.  
> **소요 시간:** 1~2시간  
> **산출물:** 검증 결과 문서 (`templates/01-validation-result.md` 작성 완료)

---

## 이 단계가 필요한 이유

많은 서비스가 출시 전에 실패하는 가장 큰 원인은  
**아이디어가 나쁜 것이 아니라, 문제가 존재하지 않기 때문**이다.

누군가의 실제 불편함을 해결하는 서비스인지,  
아니면 만들고 싶어서 만드는 서비스인지를  
코딩을 시작하기 전에 반드시 확인해야 한다.

---

## Does-Solves-Who 프레임

모든 아이디어는 아래 세 가지 질문으로 검증할 수 있다.

| 질문 | 확인 내용 | 통과 기준 |
|---|---|---|
| **Does** | 이 문제가 실제로 존재하는가? | 구체적인 상황을 말할 수 있다 |
| **Solves** | 내 서비스가 그 문제를 해결하는가? | 기존 방법보다 명확하게 낫다 |
| **Who** | 이 문제를 가진 사람이 충분히 있는가? | 타겟이 너무 좁거나 모호하지 않다 |

### 적용 방법

**1. Does — 문제 구체화**

막연한 문제 표현을 구체적인 상황으로 바꾼다.

- ❌ "사람들이 산책하기 불편해한다"
- ✅ "맞벌이 부부가 강아지를 키우는데, 낮 시간에 산책을 못 시켜서 죄책감을 느끼고 강아지도 스트레스를 받는다"

문제를 구체화하지 못한다면 — 아직 아이디어 단계다.

**2. Solves — 해결책 검증**

기존 방법과 비교하여 내 서비스가 더 나은 이유를 찾는다.

- 기존 방법: 지인에게 부탁, 펫시터 앱, 동네 카페 공지
- 내 서비스의 차별점: _________________

차별점을 한 문장으로 말할 수 없다면 — 포지셔닝이 불명확하다.

**3. Who — 타겟 정의**

"누구에게나 필요한 서비스"는 아무에게도 필요 없는 서비스다.  
가장 이 문제로 고통받는 사람 한 명을 떠올려라.

- 나이, 직업, 생활 패턴
- 이 문제를 얼마나 자주, 얼마나 심하게 겪는가
- 지금 이 문제를 어떻게 해결하고 있는가

## 페르소나 생성

타겟을 정의한 후, Claude와 함께 2~3개의 페르소나를 만든다.

### 페르소나가 필요한 이유

- 추상적인 "사용자"를 구체적인 "사람"으로 만든다
- 이후 경쟁사 분석, 기능 결정, 화면 설계의 기준점이 된다
- 가상 인터뷰의 응답자가 된다

### 좋은 페르소나의 조건

단순한 인구통계 정보를 넘어, 이 사람의 **맥락**을 담아야 한다.

| 항목 | 예시 |
|---|---|
| 이름 / 나이 / 직업 | 김지수, 34세, 마케터 |
| 생활 패턴 | 출퇴근 왕복 3시간, 재택 주 2회 |
| 이 서비스와 관련된 상황 | 3살 말티즈 반려견, 혼자 키우는 중 |
| 현재 해결 방법 | 점심시간에 급히 귀가하거나, 친언니에게 부탁 |
| 핵심 불편 / 감정 | "매번 언니한테 부탁하는 게 미안하다" |

### Claude에게 페르소나 생성 요청하는 방법

`prompts/01-idea-validation-prompts.md`의 **[P-02] 페르소나 생성** 프롬프트를 사용한다.

> ⚠️ **선택 시점**  
> Claude가 2~3개의 페르소나 초안을 제시하면,  
> 제작자가 1개를 선택하거나 수정을 요청한다.  
> Claude는 제작자의 확정을 기다린다.

## 가상 인터뷰

실제 사용자를 인터뷰하기 어려운 경우,  
Claude가 페르소나 역할을 하며 가상 인터뷰를 진행한다.

### 가상 인터뷰의 목적

- 페르소나가 이 서비스를 실제로 사용할 것인가?
- 어떤 기능에 가장 큰 가치를 느끼는가?
- 사용을 막는 요인(불안, 가격, 번거로움)은 무엇인가?

### 인터뷰 진행 방식

1. Claude가 페르소나 역할을 맡는다
2. 제작자가 인터뷰어로 질문한다 (또는 Claude가 대신 진행)
3. Claude는 페르소나의 배경과 심리에 기반한 현실적인 반응을 생성한다
4. **가상 결과임을 명시**하되, 시장 지식에 근거한다

### 핵심 인터뷰 질문 5가지

1. 이런 상황(문제)을 겪어본 적이 있나요?
2. 지금은 어떻게 해결하고 있나요?
3. 기존 방법에서 가장 불편한 점은 무엇인가요?
4. 이런 서비스가 있다면 사용할 것 같나요? 왜요?
5. 사용하지 않는다면, 가장 큰 이유는 무엇인가요?

`prompts/01-idea-validation-prompts.md`의 **[P-03] 가상 인터뷰** 프롬프트를 사용한다.

## Go / No-Go 판단 기준

가상 인터뷰와 문제 검증이 끝나면 Go/No-Go를 판단한다.

### Go 조건 (모두 충족해야 한다)

- [ ] 문제를 구체적인 상황으로 표현할 수 있다
- [ ] 타겟 페르소나가 이 문제를 실제로 겪는다고 판단된다
- [ ] 기존 해결 방법이 있지만, 내 서비스가 명확하게 더 낫다
- [ ] 가상 인터뷰에서 2명 이상이 "사용할 것 같다"고 반응했다
- [ ] 타겟이 너무 좁지 않다 (지역, 나이, 상황 조건이 3개 이상 겹치면 위험)

### No-Go 시 처리

No-Go가 나왔을 경우, Claude는 다음을 수행한다.

1. No-Go 원인 분석 (문제 불명확 / 타겟 과소 / 기존 대안이 충분 / 기타)
2. 3가지 방향 제시 + 1가지 추천:
   - **방향 A:** 타겟을 좁혀서 더 깊은 문제를 찾는다
   - **방향 B:** 문제는 유지하되 해결 방식을 바꾼다
   - **방향 C:** 처음부터 다른 아이디어로 시작한다
3. 제작자가 선택하면 해당 방향으로 재시작
