# 3단계: 서비스 기획 가이드

> **목표:** "무엇을 어떻게 만들 것인가?"를 결정한다.  
> **소요 시간:** 1~2시간  
> **선행 조건:** 2단계 경쟁사 분석표 완성 + 포지셔닝 확정  
> **산출물:** 서비스 기획서 (`templates/03-service-plan.md` 작성 완료)

---

## 이 단계가 필요한 이유

포지셔닝이 확정되면 "우리가 무엇을 만들어야 하는지"의 방향은 정해진 것이다.  
하지만 방향이 있다고 해서 바로 PRD를 쓸 수 있는 건 아니다.

3단계는 그 방향을 **구체적인 서비스**로 만드는 과정이다.

- 어떤 기능을 만들고, 어떤 기능은 버릴 것인가
- 사용자는 어떤 흐름으로 서비스를 경험하는가
- 이 서비스가 성공했다는 것을 어떻게 알 수 있는가

이 세 가지를 결정하면 4단계(문서 완성)에서 PRD를 빠르게 작성할 수 있다.

---

## 서비스 컨셉 도출

### 컨셉이란 무엇인가

컨셉은 "이 서비스는 한마디로 ___다"를 채울 수 있는 문장이다.  
포지셔닝보다 더 구체적이고, 기능 목록보다는 더 추상적인 수준이다.

좋은 컨셉은 다음 조건을 만족한다:

- 타겟이 듣자마자 "나를 위한 서비스"라고 느낀다
- 팀이 기능을 결정할 때 기준이 된다
- 경쟁사와 명확하게 구별된다

### 컨셉 도출 방법

2단계의 포지셔닝 문장을 출발점으로 삼아,  
아래 두 가지를 추가로 결정하면 컨셉이 완성된다.

**① 서비스의 핵심 가치 (Core Value)**  
사용자가 이 서비스를 쓰고 나서 얻는 가장 중요한 것 하나.

- 시간 절약인가
- 감정적 안도감인가
- 새로운 연결인가
- 경제적 이익인가

**② 서비스의 톤 앤 매너**  
사용자가 이 서비스를 어떻게 느끼기를 원하는가.

| 형용사 쌍 | 선택 |
|---|---|
| 전문적인 ↔ 친근한 | |
| 빠른 ↔ 신중한 | |
| 심플한 ↔ 풍부한 | |
| 공식적인 ↔ 캐주얼한 | |

> ⚠️ **선택 시점**  
> Claude가 서비스 컨셉 방향 3가지를 제시한다.  
> 제작자가 1개를 선택하거나 수정 후 확정한다.  
> 확정 전까지 Claude는 다음 단계로 넘어가지 않는다.

## 기능 범위 결정 (MoSCoW)

컨셉이 확정되면 기능 목록을 만들고, 각 기능의 우선순위를 정한다.  
MoSCoW 방법을 사용한다.

### MoSCoW란

| 구분 | 의미 | 기준 |
|---|---|---|
| **Must Have** | 반드시 있어야 한다 | 이게 없으면 서비스가 성립하지 않는다 |
| **Should Have** | 있어야 하지만, 없어도 출시는 가능 | 사용자 경험을 크게 높이는 기능 |
| **Could Have** | 있으면 좋다 | 리소스가 남을 때 추가 |
| **Won't Have** | 이번엔 안 한다 | 명시적으로 포기. 나중을 위한 백로그 |

### MVP 범위 설정 원칙

Must Have만으로 서비스가 작동해야 한다.  
바이브 코딩으로 만들 MVP 기준:

- **Must Have는 최대 5개** — 그 이상은 범위가 너무 넓다
- Should Have는 2차 개발로 미룬다
- "당연히 있어야 하는 것"도 Must에 들어갈 수 있다 (예: 로그인)
- 기능 하나 = 사용자가 완료할 수 있는 하나의 행동

> ⚠️ **선택 시점**  
> Claude가 기능 후보 목록을 MoSCoW로 분류해서 제시한다.  
> 제작자가 Must Have 범위를 확정한다.  
> 이 결정이 4단계 PRD의 핵심 기능 목록이 된다.

## 화면 흐름 정리

기능이 확정되면, 사용자가 실제로 서비스를 어떻게 경험하는지 흐름을 그린다.

### 화면 흐름이란

앱이나 웹서비스에서 사용자가 거치는 화면의 순서.  
설계도나 와이어프레임이 아니라, **텍스트로 표현한 흐름도**다.

### 작성 방법

1. 사용자 유형별로 주요 흐름(Happy Path)을 정한다
2. 각 흐름을 화면 이름 → 화면 이름으로 연결한다
3. 분기점(로그인 여부, 상태 변화 등)을 표시한다

**작성 예시 — 반려견 산책 매칭 앱 (의뢰자 흐름)**

```
앱 시작
  └─ 홈 화면
       └─ 산책 요청 버튼 탭
            └─ 날짜/시간 선택
                 └─ 도우미 목록 (위치 기반)
                      └─ 도우미 프로필
                           └─ 요청 확인 화면
                                └─ 결제
                                     └─ 매칭 완료 화면
```

### 화면 흐름 작성 시 확인할 것

- 신규 사용자의 첫 진입 경로가 자연스러운가
- 핵심 행동(핵심 기능 사용)까지의 클릭 수가 최소화되었는가
- 오류 상황(결제 실패, 매칭 없음 등)은 어떻게 처리하는가

## 성공 지표 설정

서비스가 잘 되고 있는지 어떻게 알 수 있을까?  
출시 전에 성공의 기준을 숫자로 정해야 한다.

### 지표의 두 가지 종류

**① 핵심 지표 (North Star Metric)**  
이 서비스의 본질적인 가치를 가장 잘 나타내는 지표 1개.

| 서비스 유형 | North Star 예시 |
|---|---|
| 매칭 플랫폼 | 월간 완료된 매칭 수 |
| 콘텐츠 서비스 | 주간 활성 사용자(WAU) |
| 커머스 | 월간 재구매율 |

**② MVP 검증 지표**  
초기 출시 후 "계속 진행해도 되는가"를 판단하는 기준.  
출시 후 4~8주 안에 달성 가능한 수준으로 설정한다.

**작성 예시:**

| 지표 | 목표 | 측정 방법 |
|---|---|---|
| 가입 후 첫 이용 전환율 | 50% 이상 | 가입 수 / 첫 이용 수 |
| 2회 이용률 | 30% 이상 | 첫 이용자 중 2회 이용자 비율 |
| 의뢰자 만족도 | 4.0점 이상 | 이용 후 별점 평균 |
