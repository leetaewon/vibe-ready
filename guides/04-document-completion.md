# 4단계: 문서 완성 가이드

> **목표:** "바로 코딩을 시작할 수 있는가?"를 확인한다.  
> **소요 시간:** 1~2시간  
> **선행 조건:** 3단계 서비스 기획서 완성  
> **산출물:** PRD + 액션플랜 (바이브 코딩 도구에 바로 사용 가능)

---

## 이 단계가 필요한 이유

3단계까지 완성된 서비스 기획서는 **제작자를 위한 문서**다.  
4단계에서 만드는 PRD와 액션플랜은 **Claude(AI 코딩 도구)를 위한 문서**다.

같은 내용이라도 AI가 읽고 실행하기 좋은 형식이 따로 있다.  
PRD는 "무엇을 만드는가"를 명확하게 전달하고,  
액션플랜은 "어떤 순서로 만드는가"를 Phase 단위로 쪼개서 지시한다.

이 두 문서가 완성되면 바이브 코딩 도구에 첨부하고 바로 빌드를 시작할 수 있다.

---

## PRD 작성 방법

PRD(Product Requirements Document)는 AI 코딩 도구가  
서비스를 정확하게 이해하고 구현하기 위한 명세서다.

### PRD의 8가지 섹션

| 섹션 | 내용 | 작성 기준 |
|---|---|---|
| **1. 앱 개요** | 서비스 목적을 2~3문장으로 | 처음 보는 사람도 이해할 수 있게 |
| **2. 타겟 사용자** | 누구를 위한 서비스인가 | 페르소나 기반으로 구체적으로 |
| **3. 핵심 기능** | Must Have 기능 목록 | 번호 목록, 기능당 1~2줄 설명 |
| **4. 제외 기능** | Won't Have 목록 | 이유 포함, AI가 임의로 추가하지 않도록 |
| **5. 화면 구성** | 전체 화면 목록과 각 화면의 역할 | 페이지/화면 단위로 구체적으로 |
| **6. 데이터 구조** | 저장해야 할 정보 | 테이블명과 주요 필드 수준으로 |
| **7. 기술 스택** | 사용할 기술과 외부 서비스 | 확정된 것만, 불확실하면 비워둬도 됨 |
| **8. 디자인 가이드** | 색상, 폰트, 분위기 | 톤 앤 매너와 참고 레퍼런스 |

### PRD 작성 원칙

**구체적이되 간결하게**  
AI는 모호한 지시에 임의로 해석해서 구현한다.  
"사용자 친화적인 UI"보다 "카드 형태의 목록, 탭 기반 네비게이션"이 낫다.

**제외 기능을 반드시 명시한다**  
"로그인은 소셜 로그인만, 이메일 회원가입은 이번엔 제외"처럼  
만들지 않을 것을 명확하게 써야 AI가 임의로 추가하지 않는다.

**데이터 구조는 상세하지 않아도 된다**  
ERD 수준이 아니라, 어떤 정보를 저장해야 하는지 목록 수준으로 충분하다.

## 액션플랜 작성 방법

액션플랜은 PRD를 기반으로 AI 코딩 도구가  
**어떤 순서로 구현할지**를 Phase 단위로 정리한 실행 계획이다.

### 표준 Phase 구조

| Phase | 내용 | 비고 |
|---|---|---|
| **Phase 1** | 프로젝트 초기 설정 | 폴더 구조, 패키지 설치, 환경 변수 |
| **Phase 2** | 데이터 구조 설계 | DB 스키마, 모델 정의 |
| **Phase 3** | UI 컴포넌트 구현 | 공통 컴포넌트, 레이아웃 |
| **Phase 4** | 핵심 기능 구현 | Must Have 기능 순서대로 |
| **Phase 5** | 테스트 및 배포 | 기본 테스트, 배포 설정 |

### 각 Phase는 체크리스트로 작성한다

AI 코딩 도구는 Phase가 끝날 때마다 제작자에게 확인을 받고 다음으로 넘어간다.  
체크리스트 형식이어야 완료 여부를 명확하게 알 수 있다.

**작성 예시:**

```
## Phase 3. UI 컴포넌트 구현

- [ ] 공통 레이아웃 컴포넌트 생성 (Header, Footer, Nav)
- [ ] 버튼 컴포넌트 (Primary, Secondary, Disabled 상태 포함)
- [ ] 카드 컴포넌트 (도우미 프로필 카드)
- [ ] 입력 폼 컴포넌트 (날짜 선택, 텍스트 입력)
- [ ] 로딩 / 에러 상태 컴포넌트
```

### 핵심 기능 구현 순서 원칙

Phase 4에서 기능 구현 순서는 다음 기준으로 정한다:

1. **의존성 먼저** — 다른 기능이 필요로 하는 기능을 먼저 만든다 (예: 인증 → 프로필 → 매칭)
2. **사용자 흐름 순서** — Happy Path의 첫 화면부터 마지막 화면 순서로
3. **리스크 높은 것 먼저** — 외부 API 연동, 결제 등 불확실한 것을 앞에 배치

## 바이브 코딩 플랫폼 선택

PRD와 액션플랜이 완성되면, 어떤 도구로 빌드할지 결정한다.

> ⚠️ **선택 시점**  
> Claude가 상황에 맞는 플랫폼 3가지를 제시한다.  
> 제작자가 1개를 선택하면 해당 플랫폼용 시작 프롬프트를 생성한다.

### 플랫폼 선택 기준

| 기준 | 확인 내용 |
|---|---|
| **서비스 복잡도** | 단순 프로토타입인가, 실제 출시용인가 |
| **기술 스택** | 프론트엔드 중심인가, 풀스택인가 |
| **배포 방식** | 링크 공유인가, 도메인 배포인가 |
| **제작자 경험** | 코드를 전혀 모르는가, 어느 정도 아는가 |

### 주요 플랫폼 비교

| 플랫폼 | 적합한 경우 | 특징 |
|---|---|---|
| **Google AI Studio (Antigravity)** | 빠른 프로토타입, 링크 공유 | 설정 없이 바로 시작, 복잡한 기능 한계 |
| **Cursor + Claude** | 실제 출시용, 풀스택 | 코드 수준 제어 가능, 로컬 환경 필요 |
| **Bolt.new** | 프론트엔드 중심, 빠른 배포 | Vite 기반, Netlify 배포 연동 |
| **Replit** | 서버 포함, 공유 쉬움 | 브라우저에서 바로 실행, 협업 가능 |
| **Lovable** | 비개발자, UI 중심 | 자연어로 수정, GitHub 연동 |

## 설정 파일 생성

바이브 코딩 도구에 따라 필요한 설정 파일이 다르다.  
Claude가 선택된 플랫폼에 맞는 설정 파일을 생성한다.

| 파일 | 용도 | 필요한 플랫폼 |
|---|---|---|
| `CLAUDE.md` | 프로젝트별 Claude 지시사항 | Cursor, 모든 Claude 기반 도구 |
| `.env.example` | 환경 변수 목록 | 외부 API 연동이 있을 경우 |
| `README.md` | 프로젝트 개요 및 실행 방법 | 모든 플랫폼 |

---

## QA 체크리스트

PRD와 액션플랜을 빌드에 넣기 전에 아래 항목을 확인한다.

| 항목 | 확인 |
|---|---|
| MVP 범위가 적절한가? (너무 많거나 적지 않은가) | ✅ / ❌ |
| 선택한 플랫폼으로 구현 가능한 수준인가? | ✅ / ❌ |
| 기술 스택이 기능 요구사항에 맞는가? | ✅ / ❌ |
| 빠진 항목이나 서로 모순된 내용이 없는가? | ✅ / ❌ |
| 액션플랜의 Phase 순서가 논리적인가? | ✅ / ❌ |
| 제외 기능이 PRD에 명시되어 있는가? | ✅ / ❌ |

> 선택한 플랫폼에서 바이브 코딩을 시작한다.
