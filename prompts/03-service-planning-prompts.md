## [P-01] 서비스 컨셉 도출

> **목적:** 포지셔닝을 구체적인 서비스 컨셉으로 만든다.

```
2단계에서 확정한 내용이야:
- 포지셔닝: [확정된 포지셔닝 문장]
- 핵심 타겟: [1단계에서 확정]
- 우리가 공략할 약점: [2단계에서 정리한 경쟁사 공통 약점]

이 내용을 바탕으로 서비스 컨셉 방향 3가지를 제안해줘.

각 방향마다:
- 컨셉 이름 (한 줄 키워드)
- "이 서비스는 한마디로 ___다" 문장
- 핵심 가치 (사용자가 얻는 가장 중요한 것 하나)
- 톤 앤 매너 (어떻게 느끼게 할 것인가)
- 이 컨셉이 포지셔닝과 어떻게 연결되는지

3가지 제안 후 1가지를 추천하고, 명확한 이유를 설명해줘.
내가 확정할 때까지 다음 단계로 넘어가지 말아줘.
```

---

## [P-02] 기능 목록 & MoSCoW 분류

> **목적:** 만들 기능과 만들지 않을 기능을 명확하게 나눈다.

```
확정된 서비스 컨셉: [P-01에서 확정된 컨셉]
핵심 타겟: [1단계에서 확정]
포지셔닝: [2단계에서 확정]

이 서비스에 필요한 기능 목록을 만들고 MoSCoW로 분류해줘.

기준:
- Must Have: 이게 없으면 서비스가 성립하지 않는다 (최대 5개)
- Should Have: 있어야 하지만, 없어도 출시는 가능하다
- Could Have: 있으면 좋지만 나중에 해도 된다
- Won't Have: 이번엔 명시적으로 포기한다

각 기능마다:
- 기능 이름
- 한 줄 설명
- 분류 이유 (Must에 넣은 이유가 특히 중요)

분류 후:
- Must Have가 5개를 넘으면 다시 줄여줘. 이유와 함께.
- Must Have 목록을 최종 확인해줘.

내가 범위를 확정할 때까지 다음 단계로 넘어가지 말아줘.
```

## [P-03] 화면 흐름 정리

> **목적:** 사용자가 서비스를 경험하는 흐름을 텍스트로 그린다.

```
확정된 Must Have 기능: [P-02에서 확정된 기능 목록]
사용자 유형: [예: 의뢰자 / 도우미 / 관리자]

각 사용자 유형별로 핵심 흐름(Happy Path)을 텍스트 흐름도로 그려줘.

포함할 내용:
- 앱/웹 시작부터 핵심 행동 완료까지의 화면 순서
- 주요 분기점 (로그인 여부, 상태 변화 등)
- 각 화면의 핵심 요소 한 줄 설명

흐름도 작성 후:
- 핵심 행동까지 클릭 수가 적절한지 확인해줘
- 빠진 화면이나 어색한 흐름이 있으면 지적해줘
- 전체 화면 목록을 정리해줘 (화면 이름 + 한 줄 설명)
```

## [P-04] 성공 지표 설정

> **목적:** 서비스 성공을 측정할 기준을 숫자로 정한다.

```
서비스 개요:
- 컨셉: [확정된 컨셉]
- 핵심 타겟: [확정된 타겟]
- Must Have 기능: [확정된 기능 목록]

이 서비스의 성공 지표를 설정해줘.

1. North Star Metric 1개
   - 이 서비스의 본질적 가치를 가장 잘 나타내는 지표
   - 왜 이 지표를 선택했는지 이유 포함

2. MVP 검증 지표 3~5개
   - 출시 후 4~8주 안에 확인 가능한 것
   - 각 지표마다: 지표명 / 목표값 / 측정 방법
   - 현실적으로 달성 가능한 수준으로 설정

3. 포기해도 되는 지표
   - 초기에는 신경 쓰지 않아도 되는 것과 이유
```

## [P-05] 서비스 기획서 완성

> **목적:** 이 단계의 모든 결정사항을 기획서 형태로 정리한다.

```
3단계에서 확정한 내용 전체를 서비스 기획서로 정리해줘.

포함할 내용:
1. 서비스 개요 (컨셉 + 포지셔닝 + 핵심 가치)
2. 타겟 사용자 (확정된 페르소나 요약)
3. 핵심 기능 범위 (Must Have 목록)
4. 제외 기능 (Won't Have 이유 포함)
5. 화면 구성 (전체 화면 목록 + 흐름)
6. 성공 지표 (North Star + MVP 검증 지표)
7. 제약 사항 (기술, 예산, 일정 등 알고 있는 것)

templates/03-service-plan.md 형식에 맞춰서 작성해줘.
이 문서가 4단계 PRD 작성의 입력값이 된다.
```
